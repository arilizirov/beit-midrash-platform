# Clean Code — notes (original summary)

> The kit's own paraphrase of this book's key principles, to guide judgment.
> This is NOT the book's text. For depth, read the book itself.

Robert C. Martin's argument: code is read far more than written, so optimise for the reader.

- Names should reveal intent; a name that needs a comment has failed.
- Functions: small, do one thing, one level of abstraction, few arguments.
- Prefer expressing intent in code over explanatory comments; comments rot, code doesn't lie.
- Remove duplication; it's the root of most maintenance pain.
- Leave code cleaner than you found it (the boy-scout rule).

*Contested in places (some advice is dated/dogmatic), so treat as a strong default, not law.*
